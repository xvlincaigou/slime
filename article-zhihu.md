# 我用 Claude 写了个"代码学习助手"：从此不再看了就忘

> 面对十万行代码的 AI 训练框架，我是如何做到一个月吃透核心架构的？

## 故事的开头

两个月前，我接到了一个任务：深入理解 [Slime](https://github.com/xvlincaigou/slime) 这个分布式 RL 训练框架的代码。

这 codebase 有多大呢？

- 核心代码超过 **10 万行**
- 涉及 **Ray、Megatron、SGLang、PyTorch** 等多个复杂系统
- 文档分散，很多设计决策藏在 PR 和 Issue 里

起初我就像大多数人一样：
- 打开 IDE，开始读代码
- 看到不懂的地方，到处打断点、翻文档
- 当时觉得"理解了"，过一周就忘得一干二净
- 再回来读，又得从头开始理解...

**这种"看代码-忘记-再看"的循环，我受够了。**

于是我想：既然有代码助手能帮我写代码，为什么不能有个"学习助手"来帮我学代码？

## 解决方案：结构化学习工作流

我的思路很简单：

**每一次和 AI 的对话，都应该被记录下来，成为可检索、可复习的知识库。**

具体来说，我设计了一个工作流：

```
提问 → AI 回答 → 自动归档 → 主题整理 → 定期复习
```

## 仓库结构长什么样？

我在项目里创建了一个 `learn/` 文件夹：

```
learn/
├── README.md              # 学习进度总览
├── questions/             # Q&A 记录（自动归档）
│   ├── Q001-training-workflow-retool.md
│   ├── Q002-placement-groups-allocation.md
│   ├── ...
│   └── Q017-train-function-model-py.md
├── topics/                # 按主题整理的知识
│   ├── training-workflow.md
│   ├── placement-group.md
│   ├── rollout-manager.md
│   └── ...
└── summaries/             # 阶段性总结（待补充）
```

### 核心设计：问答格式

每个 Q&A 文件都采用统一的格式：

```markdown
---
date: 2026-04-13
question_id: Q017
topics: ["training", "megatron", "pipeline-parallelism"]
related_files:
  - slime/backends/megatron_utils/model.py
---

# Question

讲解一下 model.py 中的 train() 函数。

# Answer

## 一句话总结
train 函数是 Megatron 训练流程的最外层循环...

## 逐行详解
...

## Key Points
- Pipeline 并行配置
- Forward Pre-hook 管理
- 日志聚合策略

## Code References
- model.py:489 - train() 函数定义
- model.py:299 - train_one_step() 函数

## Follow-up Questions
- [ ] train_one_step 内部的具体流程是怎样的？
```

这种格式的好处：
1. **问题驱动**：每个知识点都对应一个具体问题
2. **快速回顾**：开头的一句话总结，5 秒回忆起核心概念
3. **深度可查**：逐行详解 + 代码引用，需要细读时能深挖
4. **知识网络**：topics 和 related_files 建立关联

## 实际效果如何？

一个月下来，我记录了 **17 个深度问答**，覆盖了：

| 主题 | 问题数 | 核心收获 |
|------|--------|----------|
| Ray 资源调度 | 4 | 彻底搞懂了 Placement Group 的分配策略 |
| SGLang 推理引擎 | 3 | 理解了 PD 分离、故障容错机制 |
| Megatron 训练 | 7 | 掌握了 Pipeline/Torch/Context 三种并行 |
| 数据加载 | 2 | 明白了动态 batch size 和序列长度均衡 |

**最有价值的是**：当我第三周回头看第一周的笔记时，那些"一句话总结"帮我瞬间找回了记忆。再也不用重新读代码了。

## 具体怎么用？

### 对于学习者

1. **Fork 或 clone 你的目标仓库**
2. **创建 `learn/` 文件夹**，复制我的结构
3. **开始提问**：每遇到一个不懂的地方，就问 AI，让它记录到 `questions/` 文件夹
4. **定期整理**：相同主题的问题，汇总到 `topics/` 文件夹
5. **复习时只看 README**：通过索引快速定位需要回顾的内容

### 对于开发者

如果你想把这个工作流用在自己的项目里，可以：

```bash
# 1. 创建 skill 文件夹
mkdir -p .claude/skills/learn-workflow

# 2. 写入 SKILL.md（定义工作流规范）
# 内容可以参考我的仓库

# 3. 使用时直接调用
/learn-workflow 你的问题
```

AI 会自动按照规范回答问题，并创建对应的 Q&A 文件。

## 为什么这个方法有效？

**认知科学原理**：

1. **主动回忆**：问答形式比被动阅读记忆效果更好
2. **间隔重复**：通过 README 索引定期回顾，符合遗忘曲线
3. **精细编码**：代码引用 + 逐行解释，建立深度理解

**工程实践优势**：

1. **可检索**：Git 管理，随时搜索历史问题
2. **可复用**：类似项目可以直接参考已有笔记
3. **可迭代**：发现理解错误时，直接修改笔记

## 示例：我是怎么学 Placement Group 的

**我的问题**："create_placement_groups 是在做什么？分配规则是什么？"

**AI 的回答结构**：
- 一句话总结："Ray Placement Group 的 GPU 资源分配策略..."
- 分配规则详解：节点优先 vs 负载均衡
- 代码走读：从 32 行开始逐行解释
- 流程图：可视化分配过程
- 关键概念：Bundle、Placement Strategy、Strict Spread

**现在回头看**：我已经完全忘了那段代码的具体实现，但通过那篇笔记，我 30 秒就能重新理解整个机制。

## 一些经验总结

### DO
- **问具体问题**："这个函数在干什么"比"解释一下这个项目"效果好
- **要求代码引用**：一定要具体到文件名和行号
- **做一句话总结**：这是快速复习的关键
- **标记后续问题**：暂时没搞懂的，记下来以后深挖

### DON'T
- 不要只让 AI 总结，不保留细节
- 不要把所有内容堆在一个文件里
- 不要忽略 topics 整理（这是建立知识网络的关键）

## 最后的话

读代码就像读一本没有目录的书。传统的方法是一次性读完，期望能记住所有内容——这根本不现实。

我的工作流本质上是**给这本书写一个目录和读书笔记**，而且是用"问题"作为索引。当你需要回顾时，不需要重读整本书，只需要看笔记，然后针对性地翻回原书验证。

**完整的代码和模板**可以在我 Fork 的仓库里找到：
https://github.com/xvlincaigou/slime/tree/learn

如果你也想尝试这个方法，建议从一个小模块开始，创建 3-5 个 Q&A，感受一下这种结构化学习的效率。
